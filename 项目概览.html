Brufik in One

一个把 硬件 / 固件 与 语音 AI 后台 合在一个仓库里的开源「桌面陪伴机器人」项目。设备端是一只会转头、有表情、能对话的小机器人；后台端是一条完整的 VAD → ASR → LLM → TTS 语音管线，外加人脸识别、记忆、定时任务和 Web 管理控制台。两者通过一条 WebSocket 长连接（/asr_chat）实时联动。

ESP32-S3 固件 双轴舵机转头 1.83″ 表情屏 FunASR 本地识别 通义千问 LLM Doubao / Paddle TTS 人脸识别 + 跟随 GPL-3.0 / CERN-OHL

📦 项目定位

这是一台「桌搭子」级别的 AI 语音陪伴机器人，DIY 友好、成本低，强调表情/动作的拟人化反馈。

🎯 是什么

放在桌面上的小机器人，能听你说话、用自己的「声音 + 表情 + 转头动作」回应你，还能看着你（摄像头人脸跟随）。

👥 给谁用

面向开发者 / 创客 / DIY 玩家。提供完整 PCB、3D 结构件、固件与后台源码，可自行烧录、组装、私有化部署。

🧩 一体化仓库

由两个原本独立的项目合并而来：hardware/（Brufik 硬件）+ service/（opendesk-service 后台），统一管理、协同迭代。

关键设计：设备「轻」、后台「重」。ESP32 几乎只负责采音、播音、画屏、转舵机；所有智能（识别、对话、人脸、记忆、定时）都在后台。默认模式 asr_chat_device_pb_only 下，设备只需解析 pb_start / pb_chunk / pb_end 三类下行帧 + 配套的 24kHz PCM，复杂度低、固件稳定。

🤖 实体设备 HARDWARE

基于 Seeed XIAO ESP32-S3 Sense 的桌面小机器人：一块表情屏 + 双轴转头 + 麦克风/喇叭 + 摄像头。

🧠主控 & 感知

  • MCU：XIAO ESP32-S3（双核，8MB PSRAM）
  • 摄像头：板载 OV2640（120° 广角）
  • 麦克风：板载 PDM 麦（无需外接）
  • 喇叭：2011 腔体喇叭 + MAX98357A I2S 功放

🖥️显示 & 动作

  • 屏幕：1.83″ ST7789P SPI LCD，240×284
  • 转头：双轴舵机 — X 平移(60–120°) / Y 俯仰(70–110°)
  • 状态灯：可编程 RGB LED
  • 供电：USB-C 5V（舵机）+ 3.3V（逻辑）

引脚与硬件映射（deskbot_config.h）

模块接口引脚
LCD 屏SPIMOSI=GPIO9 · SCK=GPIO7 · CS=GPIO2 · DC=GPIO3
舵机 X（平移）PWMD7 / GPIO44，限位 60–120°
舵机 Y（俯仰）PWMD6 / GPIO43，限位 70–110°

机械结构提供完整 STEP 模型（Brufik step all v1.01.STP）与图文 装配说明书（说明书1.02PDF.pdf）；PCB 提供 KiCad 重建文件（mechanical/pcb/Board1/），也支持手工飞线方案。

📡 设备通信协议 SERVICE

设备与后台之间是单条持久 WebSocket（/asr_chat?device_id=&api_key=，端口 9000），采用「JSON + 二进制」配对帧，不做 base64。

⬆️ 上行（设备 → 后台）

类型含义
audio + binOpus/PCM16 音频流，flush 触发处理
camera_frameJPEG 帧（可选），触发人脸检测/跟随
pb_ack背压反馈：缓冲时长、当前舵机角度
ping保活

⬇️ 下行（后台 → 设备）

类型含义
pb_start序列首帧（含优先级 / 动作）
pb_chunk续传分片
pb_end序列结束
pb_single一帧打包全部
每个 pb 帧承载三种东西：音频（s16le 24kHz PCM）、动画（anim[]：背景/眼/嘴/鼻等图层 + 音素口型）、舵机（servo[]：绝对/相对/保持，X/Y 角度 + 时长）。画布固定 284×240。下行带优先级队列（0 空闲 / 1 说话 / 2 紧急 / 3 调试），高优先级到达可清掉旧的低优先级播放，实现「打断」。

🧠 LLM 与工具生态 SERVICE

人设是「小歪」——桌面陪伴体，回答简短（<50 字）、诚实、能感知人脸与时间。LLM 返回 JSON，可携带工具调用，后台执行后把结果喂回，循环至无工具为止。

记忆 / 会话 / 多轮

  • 会话：10 分钟空闲超时，按 session_id 存历史，下一轮带上下文。
  • 长期记忆：每设备一份 user_memory.json，注入 LLM 上下文，可在控制台增删改。
  • 人脸档案：MediaPipe 检测 + InsightFace ArcFace 512 维向量（相似度阈值 0.4）识别身份，可命名。
  • 数据隔离：每个 device_id 独立目录（人设/记忆/人脸/会话/临时文件）。

👄 TTS 与口型同步 SERVICE

这是该项目让机器人「活起来」的关键——把语音、嘴型、屏显文字三者对齐。

🔊 双 TTS 后端

PaddleSpeech（默认·本地）：ONNX 流式声码器，按音素数把 PCM 均分得到近似时长，端口 8092 旁路服务。

Doubao（豆包·云端，可选）：字节 Seed-TTS 2.0 双向 WebSocket，可返回 words/phonemes 精确时间戳；无时间戳时按拼音音节回退均分。

📝 底部屏显文字

LLM 输出可带 screen_text（支持 \n 换行）+ screen_text_color。后台渲染成 pb 动画里的文字图层，底部居中显示、自动换行。

与朗读分离：tts 是念出来的，screen_text 只显示不念。

音素口型映射：face_mouth_by_phoneme.json 把音素映射到嘴部图层形状/位置/颜色；TTS 返回的音素时间轴驱动屏幕逐帧换嘴型，从而和声音同步——这就是最近一次提交「Doubao TTS backend with phoneme lip-sync」做的事。

🖥️ Web 控制台 SERVICE

Flask 独立进程（端口 5050），邮箱+密码登录，SQLite 存储，给用户做设备绑定、用量监控、记忆/人脸管理与调试。

💡 设计思路评估

👍 亮点

  • 端云职责清晰：设备极简（只解析 3 类 pb 帧），智能全在后台，固件好维护。
  • 拟人化反馈到位：语音 + 表情 + 口型 + 转头 + 屏字五通道，音素级同步，"活"感强。
  • 可插拔：ASR/LLM/TTS 各自可换；TTS 本地(Paddle)/云(Doubao) 双后端。
  • 产品化完整：带账号体系、API Key、用量统计、记忆/人脸/定时管理与调试控制台。
  • 开源全栈：从 STEP 结构件、PCB、bootloader 到后台一应俱全，可完整复刻。

🔍 可关注点

  • 依赖外部 API：LLM 走 DashScope、Doubao 走字节云，需密钥与网络，非纯本地。
  • 资源占用：本地 ASR 模型 ~900MB + 人脸模型，后台对机器有一定要求。
  • 设备录音是定长轮次：固件按最长 10s/轮录音，长句/打断体验取决于 VAD 调参。
  • 配置偏手工：固件 WiFi/Key 为编译期写死（也支持热点配网），首次门槛偏开发者向。
